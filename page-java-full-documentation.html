Java 21 e Virtual Threads

Uma das maiores revoluções recentes do Java foi o Project Loom com Virtual Threads.

Antes do Loom, aplicações concorrentes dependiam fortemente de pools de threads do sistema operacional, gerando alto consumo de memória e limitações de escalabilidade.

Com Virtual Threads, Java agora consegue lidar com milhões de operações concorrentes utilizando threads extremamente leves. 

ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();

Isso tornou Java extremamente competitivo para aplicações:

  • High throughput
  • APIs massivas
  • Microsserviços
  • Streaming
  • Processamento concorrente

2 - JVM (Java Virtual Machine)

A JVM é o núcleo do ecossistema Java.

Ela é responsável por:

  • Gerenciamento de memória
  • Execução do bytecode
  • Garbage Collection
  • JIT Compilation
  • Thread management
  • Class loading
  • Otimizações em runtime

Fluxo interno da JVM

Class Loader

O Class Loader carrega classes dinamicamente.

Existem três principais:

  • Bootstrap ClassLoader
  • Extension ClassLoader
  • Application ClassLoader

JIT Compiler

O Just-In-Time Compiler converte bytecode em código nativo otimizado em tempo de execução.

Isso faz Java atingir performance extremamente alta em workloads longos.

4 - Virtual Threads aprofundado

Virtual Threads representam uma das maiores mudanças da história do Java.

Antes do Project Loom:

  • Cada requisição consumia thread pesada do SO
  • Alto consumo de memória
  • Context switching caro
  • Limitação de concorrência

Agora Virtual Threads são extremamente leves. 

ExecutorService executor =
    Executors.newVirtualThreadPerTaskExecutor();

executor.submit(() -> {
    System.out.println("Virtual Thread");
});

Impactos arquiteturais

  • Milhões de conexões concorrentes
  • Menos necessidade de programação reativa
  • Maior simplicidade
  • Redução de complexidade assíncrona

Quando evitar Virtual Threads

  • CPU-bound pesado
  • Bibliotecas não thread-safe
  • Drivers antigos

5 - APIs assíncronas no Java

APIs assíncronas permitem que operações demoradas não bloqueiem threads principais.

CompletableFuture

CompletableFuture.supplyAsync(() -> {
    return userService.findUser();
}).thenApply(user -> {
    return mapper.toDto(user);
});

Fluxo assíncrono

Request → Thread recebe request → Task async disparada → Thread liberada → Processamento ocorre em background → Resposta retornada posteriormente

Quando usar async

  • Integrações externas
  • IO-bound
  • Uploads
  • Mensageria
  • Streaming
  • Processamento distribuído

7 - Hibernate Internals aprofundado

Hibernate não é apenas um ORM simples.

Internamente ele possui:

  • Persistence Context
  • Dirty Checking
  • First Level Cache
  • Second Level Cache
  • Entity Manager
  • Session Factory
  • Lazy Loading

Fluxo interno Hibernate

Entity → Persistence Context → Dirty Checking → SQL Generation → JDBC → Database

Dirty Checking

Hibernate monitora objetos em memória.

Ao detectar alterações:

  • gera UPDATE automaticamente
  • sincroniza contexto
  • executa flush

Problemas enterprise clássicos

  • N+1 Queries
  • LazyInitializationException
  • Overfetching
  • Queries escondidas
  • Alto uso de memória

20 - Arquivo de start e como iniciar uma aplicação Java

Aplicação Java simples (sem Spring)

O ponto de entrada é uma classe com método public static void main(String[] args). 

public class Main {
    public static void main(String[] args) {
        System.out.println("Hello, Java!");
    }
}

Para iniciar: 

javac Main.java
java Main

Aplicação Spring Boot

O ponto de entrada é a classe anotada com @SpringBootApplication. 

@SpringBootApplication
public class DemoApplication {

    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}

Para iniciar via Maven: 

mvn spring-boot:run

Para iniciar via JAR: 

mvn clean package
java -jar target/demo-0.0.1-SNAPSHOT.jar

22 - Como debugar com IntelliJ IDEA

  1. Abra o projeto no IntelliJ IDEA.
  2. Coloque um breakpoint clicando à esquerda da linha de código.
  3. Execute a aplicação em modo Debug:
    • Para Spring Boot: botão “Debug” na configuração de execução.
  4. Quando o código atingir o breakpoint, o IDE irá pausar a execução.
  5. Use:
    • Step Over (F8)
    • Step Into (F7)
    • Step Out (Shift+F8)
    • Evaluate Expression para inspecionar variáveis

Também é possível debugar remotamente configurando a JVM com: 

-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005

25 - ORMs e JDBC: qual o melhor, mais utilizados e diferenças

ORMs

  • Hibernate / JPA – padrão de mercado, abstrai SQL, mapeia entidades.
  • EclipseLink – outra implementação JPA.

JDBC puro

Uso direto de java.sql.Connection, PreparedStatement, ResultSet. Máximo controle, porém mais verboso.

Abordagens intermediárias

  • Spring JdbcTemplate / NamedParameterJdbcTemplate – simplifica JDBC, mantém SQL explícito.
  • JDBI – camada leve sobre JDBC com mapeamento mais amigável.

Em sistemas legados, Hibernate/JPA é extremamente comum. Para evitar complexidade de ORMs pesados, JDBI e JdbcTemplate são ótimas opções, mantendo performance e controle sobre o SQL.

26 - Hibernate, JDBI e Spring JdbcTemplate / NamedParameterJdbcTemplate

Hibernate

É um ORM completo que implementa JPA. Ele:

  • Mapeia classes Java para tabelas
  • Gera SQL automaticamente
  • Gerencia cache, lazy loading, dirty checking

Mais utilizado em sistemas legados e em aplicações enterprise que priorizam produtividade sobre controle fino de SQL.

JDBI (Jdbi3)

Biblioteca que fica entre JDBC puro e um ORM:

  • Você escreve SQL manualmente
  • JDBI mapeia resultados para objetos
  • Menos mágica que um ORM, mais controle

Spring JdbcTemplate / NamedParameterJdbcTemplate

Abstrações do Spring sobre JDBC:

  • Gerenciam recursos (Connection, PreparedStatement)
  • Reduzem boilerplate
  • Permitem SQL explícito
  • NamedParameterJdbcTemplate permite parâmetros nomeados

Qual o mais utilizado em sistemas legados?

  • Hibernate/JPA é o mais comum em legados enterprise.

Qual usar para evitar complexidade de ORM pesado, com performance e sem esconder SQL?

  • Spring JdbcTemplate ou JDBI são excelentes escolhas.

Qual o mais indicado para Spring Boot?

  • Spring Data JPA (Hibernate) para produtividade.
  • Spring JdbcTemplate para cenários de alta performance e SQL complexo.

Facilidade de implementação

  • Mais fácil: Spring Data JPA
  • Intermediário: JdbcTemplate
  • Mais flexível: JDBI

Tabela comparativa

Ferramenta Tipo Controle sobre SQL Complexidade Uso típico
Hibernate / JPA ORM completo Baixo (SQL gerado) Média/Alta Enterprise, legados, produtividade
Spring Data JPA Abstração sobre JPA Baixo Baixa/Média APIs rápidas, CRUD
Spring JdbcTemplate Abstração JDBC Alto (SQL explícito) Média Financeiro, queries complexas
NamedParameterJdbcTemplate Abstração JDBC Alto Média SQL com muitos parâmetros
JDBI Camada leve sobre JDBC Alto Média Serviços que querem controle e menos boilerplate

28 - Principais modificadores de acesso

  • public – acessível de qualquer lugar

  • private – acessível apenas dentro da própria classe

29 - Interfaces, injeção de dependência, inversão de dependência e SOLID

Interfaces em Java

Interfaces definem contratos: 

public interface PaymentGateway {
    void processPayment(double amount);
}

Injeção de dependência (DI) e inversão de controle (IoC)

Em vez de a classe criar suas dependências, elas são injetadas: 

@Service
public class OrderService {

    private final PaymentGateway paymentGateway;

    public OrderService(PaymentGateway paymentGateway) {
        this.paymentGateway = paymentGateway;
    }
}

O Spring gerencia as instâncias e injeta a implementação correta.

SOLID no Java

  • S – Single Responsibility: cada classe com uma responsabilidade clara.
  • O – Open/Closed: aberto para extensão, fechado para modificação.
  • L – Liskov Substitution: subclasses devem poder substituir a superclasse.
  • I – Interface Segregation: interfaces específicas, não gigantes.
  • D – Dependency Inversion: dependa de abstrações, não de implementações.

37 - APIs REST no Java

Exemplo com Spring Boot

@RestController
@RequestMapping("/api/users")
public class UserController {

    private final UserService service;

    public UserController(UserService service) {
        this.service = service;
    }

    @GetMapping("/{id}")
    public ResponseEntity<UserDto> findById(@PathVariable Long id) {
        return ResponseEntity.ok(service.findById(id));
    }

    @PostMapping
    public ResponseEntity<UserDto> create(@Valid @RequestBody UserRequest request) {
        return ResponseEntity.status(HttpStatus.CREATED).body(service.create(request));
    }
}

38 - gRPC no Java

Definição do serviço (.proto)

syntax = "proto3";

service UserService {
  rpc GetUser (GetUserRequest) returns (GetUserResponse);
}

message GetUserRequest {
  int64 id = 1;
}

message GetUserResponse {
  int64 id = 1;
  string name = 2;
}

Implementação em Java (servidor)

public class UserServiceImpl extends UserServiceGrpc.UserServiceImplBase {

    @Override
    public void getUser(GetUserRequest request,
                        StreamObserver<GetUserResponse> responseObserver) {
        GetUserResponse response = GetUserResponse.newBuilder()
            .setId(request.getId())
            .setName("User " + request.getId())
            .build();

        responseObserver.onNext(response);
        responseObserver.onCompleted();
    }
}

39 - Testes unitários e de integração

@ExtendWith(MockitoExtension.class)
class UserServiceTest {

40 - Clean Code em Java

Boas práticas

  • Nomes claros para classes, métodos e variáveis
  • Métodos curtos, com uma responsabilidade
  • Evitar parâmetros demais (usar objetos de request)
  • Evitar comentários desnecessários – código deve ser autoexplicativo
  • Separar camadas (controller, service, repository)

O que evitar

  • Métodos gigantes
  • Classes “Deus” que fazem tudo
  • Abusar de static
  • Duplicação de código

41 - Arquiteturas e abordagens para microserviços robustos sem complexidade excessiva

Abordagem recomendada

  • Arquitetura em camadas simples (Controller → Service → Repository)
  • Banco por serviço (quando possível)
  • Comunicação síncrona REST e assíncrona via mensageria (Kafka, RabbitMQ)
  • Configuração centralizada (Spring Cloud Config ou equivalente)
  • Observabilidade desde o início

Evitar

  • Microserviços demais sem necessidade
  • Complexidade prematura (CQRS/Event Sourcing sem real demanda)
